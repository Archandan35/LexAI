3. Problems Found
Critical:
- No real backend server — This is a client-only SPA. The browser talks directly to Supabase/MongoDB/Firebase REST APIs using anon keys. This means:
- All secrets exposed to the browser (mitigated by clientSecretGuard.js)
- No server-side validation of business logic
- No server-side logging/auditing of DB operations
- Backup runs in the browser (memory-bound: can't handle >100MB DB)
- File uploads go directly from browser to storage
- Anyone with browser DevTools can inspect all API calls, schema, data flows
- Single CSS file — 26,675 lines in styles/index.css. No CSS modules, no CSS-in-JS, no component-scoped styles. This is not maintainable at enterprise scale. Class name collisions, dead CSS, and specificity battles are guaranteed.
High:
- No TypeScript — Plain JSX means no compile-time type checking, no interface enforcement, no refactoring safety. Especially dangerous in a 70+ page, 49-repository, 50-schema application.
- No testing infrastructure — No test files found. No Jest, Vitest, Playwright, or Cypress config.
- 25 MB max upload with NO compression — The upload pipeline (storageService.uploadDocument) passes raw files to storage with zero client-side compression. Images, PDFs, and documents are stored as-is.
- Query cache is in-process memory — queryCache.js is a Map in memory. If the user navigates away and back, or if the browser tab is refreshed, all cache is lost and re-fetched.
- Search is in-memory TF only — LocalSearchProvider.js does simple term-frequency scoring on a flat array. No persistent index, no ranking, no faceted search.
Medium:
- ID generation is hybrid client/server — IDEngine.js tries next_lx_id RPC first, falls back to client-side _sequences collection, then to in-memory fallback. Race conditions possible in multi-tab scenarios.
- No request deduplication in query cache — Two components mounting simultaneously with the same key will fire two identical requests.
- React Context as state management — AuthContext, AppDataContext, SettingsContext, ToastContext, DebugContext, FABContext — contexts re-render all children on any change.
- No React.memo, useMemo, useCallback usage in most components — leads to unnecessary re-renders.
- exec_sql RPC granted to anon role — The Supabase database provider grants exec_sql to the anon role (unauthenticated users). While safe_ddl blocklists destructive operations, exec_sql can run arbitrary SELECT queries. This is a design compromise for the browser-only architecture.

7. Key Architectural Smells
1. exec_sql exposed to anon role (SchemaCompiler.js:378) — This is required because the SPA has no backend, but it means any website visitor can run SELECT queries against the Supabase database. Mitigated by RLS, but still an unnecessary attack surface.
2. _sequences collection with no locking (IDEngine.js:108-118) — Multi-tab browser usage can produce duplicate IDs because get + update is not atomic on localStorage.
3. ensureCollectionExists called on every write failure (baseRepository.js:135-142) — The repository retries every failed write by first attempting to create the table. This adds latency and complexity. Should only provision on first-write detection, not on every transient failure.
4. Anonymous auth.uid() RLS fallback (SchemaCompiler.js:120-148) — The current_user_role() function silently falls back to 'user' when the users table is inaccessible. This means an unauthenticated request can be treated as 'user' role under error conditions.
5. No request timeout for local provider — The abstract DatabaseProvider has no timeout, retry, or circuit-breaker. If localStorage is corrupted, calls hang.

Pass 2: Database, Schema, Data Layer, Search & RLS Audit
1. Current Database Architecture
Provider architecture:
DatabaseProvider (abstract)         ← Core CRUD contract
  ├── LocalDatabaseProvider          ← localStorage adapter
  ├── SupabaseDatabaseProvider       ← PostgREST REST API
  ├── MongoDatabaseProvider          ← MongoDB Data API
  └── FirebaseDatabaseProvider       ← Firestore REST API
The provider is selected via VITE_DATABASE_PROVIDER env var. The factory pattern in providers/database/index.js constructs a singleton. Every data operation flows through:
Service → Repository → FieldMapper → DatabaseProvider → Remote API
Schema system:
- 50+ entity schemas in data-provider/schema/*.schema.js
- Each schema defines: collection, fields (name → type), required, defaults, indexes, relations, primaryKey
- SchemaCompiler.js generates provider-specific DDL (PostgreSQL, MongoDB, Firebase rules, localStorage)
- SchemaDiffEngine.js compares canonical schema against live backend (table, column, index, type diffs)
- EntityRegistry.js maps entity names ↔ provider table names
- FieldMapper.js maps canonical field names ↔ provider column names
2. Problems
Critical:
1. exec_sql granted to anon role (SchemaCompiler.js:378) — The exec_sql RPC function is granted to both authenticated and anon PostgreSQL roles. This means any unauthenticated website visitor can run arbitrary SQL against the Supabase database. The safe_ddl function has a blocklist, but exec_sql itself has no restriction — it executes any SQL including recursive CTEs, large JOINs, or CPU-intensive queries that could cause denial of service.
2. No Row-Level Security (RLS) on local provider — The local (localStorage) provider has zero security. Data is stored in plaintext in the browser's localStorage. Anyone with physical access to the machine, or any browser extension, can read all data.
3. No connection pooling — Every browser tab creates a separate provider instance. Supabase has a limited number of database connections. With 100+ simultaneous users, each with multiple tabs, connection exhaustion is guaranteed.
4. List queries return ALL rows — The list() method on the base provider has no mandatory pagination. The SupabaseDatabaseProvider.list() applies a default limit only if one is passed. Querying provider.list('audit_logs', {}) returns every audit log row. On a system with 10M records, this single call would transfer hundreds of MB to the browser and crash the tab.
High:
5. Browser-side database operations are single-threaded — All CRUD operations run in the browser's main thread. A bulkCreate of 10,000 records (which iterates sequentially) or a snapshot of all collections blocks the UI for seconds.
6. _sequences collection has no atomicity — IDEngine.generate() does a get() then update() on the _sequences collection. In localStorage, this is synchronous so it works. On Supabase/Mongo/Firebase, two browser tabs can read the same current value, both increment to current + 1, and produce identical IDs.
7. ensureCollectionExists on every write failure (baseRepository.js:135-142) — When a create/update fails, the repository tries to create the table and retry. This means every transient network error triggers a DDL attempt. On a production system with intermittent network, this would flood the database with redundant CREATE TABLE IF NOT EXISTS calls.
8. grantCollectionAccess creates RLS policies that allow ALL operations for anon (baseRepository.js:148-149) — The auto-provisioning path creates a policy "${table}_anon_all" that grants for all to anon using (true) with check (true). This completely bypasses RLS for any table that was auto-created.
Medium:
 9. Schema version 34 has no migration files — SCHEMA_VERSION = 34 but the migrations/ directory has only 002-case-management.sql, migration-to-v18.sql, migration-to-v19.sql, migration-to-v20.sql. There's no migration path from v18→v34. A fresh install would work (creates all tables from the canonical schema), but upgrading from a prior version would require manual SQL.
10. Indexes are not verifiable from the browser — SupabaseDatabaseProvider.listColumns() and listIndexes() return null. PostgREST doesn't expose information_schema.columns or pg_indexes. So the health engine can never verify that indexes exist on the Supabase backend, leading to silent performance degradation.
11. No composite indexes — All schema indexes are single-column (indexes: ['email', 'role_code', 'status']). Common queries like WHERE status = 'Active' AND role_code = 'admin' need a composite index (status, role_code) for efficiency.
12. No partial indexes — WHERE archived = false is a common filter. Adding CREATE INDEX ... WHERE archived = false would dramatically improve active-case queries.
13. No full-text search indexes — The cases.description, documents.text, drafts.content fields are all text or string with no GIN/tsvector index. The search service does in-memory TF scoring of all rows — O(n) scan on every query.
14. No soft-delete columns — There is no deleted_at or is_deleted field on any schema. remove() does a hard DELETE. There is no way to recover accidentally deleted records.
15. No audit timestamps on some schemas — judgments, permissions, order_sheet_templates have no created_at/updated_at fields in their schema.
16. Naming inconsistency — Some schemas use camelCase (createdAt, updatedAt), others use snake_case in the field definitions (created_at, role_code). The FieldMapper is supposed to handle the translation, but the canonical schema files mix both conventions. This causes silent translation failures when the mapper doesn't have an entry.
3. Enterprise Recommendation
Phase 1 — Immediate:
- Add mandatory pagination to DatabaseProvider.list() — enforce { limit, offset } with a configurable max page size (e.g., 1000)
- Remove exec_sql grants from anon — only authenticated users should execute arbitrary SQL
- Remove _anon_all RLS policies from grantCollectionAccess() — auto-created tables should not bypass RLS
- Add deleted_at timestamp to all schemas for soft-delete support
Phase 2 — Short-term:
- Add composite indexes for the most common query patterns
- Add GIN/tsvector indexes for full-text search on cases.description, documents.text, drafts.content
- Implement server-side pagination/cursor in the repository layer
- Add connection retry with exponential backoff (currently single-attempt)
Phase 3 — Long-term:
- Introduce a backend API server (BFF) that mediates all database access
- Remove the exec_sql RPC entirely — all DDL should go through a migration service
- Implement proper database migration files (numbered, timestamped, versioned)
- Add database-level audit triggers (PostgreSQL audit extension or logical replication)
4. Real-World Example
Imagine a bank where every teller terminal connects directly to the core banking database using a shared "teller" password. Each teller can run SQL queries, create new tables on the fly, and delete rows — all from their browser-based terminal. The bank's "security" is a sign on the wall that says "please don't delete anything important."

That's what exec_sql granted to anon + _anon_all RLS policies is like.
Ideal enterprise approach:
The bank has a server layer. Tellers use a purpose-built UI. The server validates every request, checks permissions, logs every operation, and limits what each teller can do. The teller never touches SQL. The bank's database is firewalled — only the application server can connect.

Pass 3: Authentication, RBAC, Authority Engine & Security Audit
1. Current Architecture
Authentication:
- AuthProvider (abstract) → SupabaseAuthProvider (only implementation)
- Auth flows through Supabase GoTrue REST API: signUp, signIn, signOut, getSession
- Session stored in localStorage under lexai.auth.session
- Token refresh via grant_type=refresh_token to Supabase /auth/v1/token
- No local/Mock auth provider — Supabase is the only option
Authorization / RBAC:
- Permission strings follow module.action format (e.g., manageCase.delete, roles.edit)
- 30 modules × 27 actions = 810 possible permission keys
- Roles store permissions: string[], optional all: boolean flag (super admin)
- Role inheritance via inherits: string[] (role code references)
- Per-user overrides via grants: string[] (add) and denies: string[] (subtract)
- Resolution: rbacLogic.resolve(user, roles) → { can(perm), has(module, action), isSuperuser, sourceOf(perm) }
Three-layer permission enforcement:
1. Route level: RequireAuth component wraps routes, checks canViewModule(module)
2. UI level: PermissionGate component hides buttons/elements based on can(perm)
3. Service level: permissionGuard.requireCapability(perm) throws PERMISSION_DENIED when called without authorization
Database-level security (RLS):
- lx_is_superadmin(), lx_is_manager() PostgreSQL functions that check the role's all flag and permissions in the roles table
- Policies on every table: admin_all, manager_all, user_select, anon_select
- user_role_registry table with trigger-sync from users table
- current_user_role() function maps auth.uid() → role_code from users table
2. Problems
Critical:
1. No server-side enforcement — The entire permission system runs in the browser. rbacLogic.resolve() runs in AuthContext's useMemo. permissionGuard.requireCapability() is a module-level variable set by syncPermissionGuard(). A user with browser DevTools can:
- Call requireCapability with any permission → it returns from in-memory state
- Directly call databaseService.create('cases', data) bypassing the UI entirely
- Modify activeCan in memory to always return true
- The service-layer guard is a facade — it protects against casual bypass but not against a determined user
2. Password hashing is "demo-grade" in browser (crypto.js:5) — SHA-256 of salt:password with 16-byte random salt. No PBKDF2, bcrypt, scrypt, or Argon2. The code itself says "provides DEMO-grade protection only — anyone with devtools can read memory."
3. SupabaseAuthProvider.signIn creates users with empty role (SupabaseAuthProvider.js:103-114) — When a user signs in via Supabase Auth but has no corresponding row in the application users table, it auto-creates one with roleCode: ''. This creates a user with zero permissions who can still authenticate and access public pages.
4. No brute-force protection — The Supabase /auth/v1/token?grant_type=password endpoint is called directly from the browser with no rate limiting, no CAPTCHA, no account lockout.
High:
5. Session token stored in localStorage in plaintext — The JWT access token and refresh token are stored in localStorage.getItem('lexai.auth.session') as a JSON string. Any browser extension or XSS vulnerability can steal the token. Enterprise standard: httpOnly cookie or in-memory session with short-lived tokens.
6. permissionGuard.syncPermissionGuard is a global mutable — The activeCan, activeIsSuperuser, activeUser variables are module-level mutable state. If AuthContext unmounts or the user logs out, the stale permissions remain until syncPermissionGuard() is called again. Race conditions during rapid auth changes could leak permissions.
7. execSql RPC exposed to authenticated role — The exec_sql function is granted to authenticated via SchemaCompiler.js:379. Any authenticated user (not just admins) can run exec_sql('select * from audit_logs'). The safe_ddl blocklist blocks destructive operations but does not block SELECT queries.
8. lx_is_superadmin() reads from roles table via security-definer — While correctly bypassing RLS recursion, this function queries the roles table on EVERY RLS evaluation. On a table with 50+ RLS policies and 100+ rows, a single query can trigger dozens of roles table lookups, creating a performance bottleneck.
Medium:
 9. No permission caching — Every call to can(perm) iterates through role permissions, grants, and denies sets. With 50+ roles and 810 permissions, resolution is O(n) on every call. In a component tree with 100+ PermissionGate instances (sidebar, tables, buttons), this creates measurable re-render overhead.
10. PermissionMatrix loads all 30 modules × 27 actions — The permission management UI shows 810 checkboxes. For a user with 10+ modules granted, this is an unwieldy interface. Enterprise systems group permissions by category and use role templates.
11. No permission audit trail for denials — permissionGuard.requireCapability() throws a generic PERMISSION_DENIED error but does not log the denied request to the audit log. Security teams need to know who was denied access to what.
12. all flag is binary — The all flag on a role means "can do everything." There's no concept of scoped super-admin (e.g., "can manage users but not view audit logs"). Enterprise RBAC uses hierarchical admin levels.
13. No API key authentication — The config.js reads API keys for OpenAI, Anthropic, Gemini, etc. and ships them to the browser. Anyone can extract these from the JS bundle and use them. The clientSecretGuard.js warns but does not prevent.

Immediate fixes:

- Add permission denial audit logging to requireCapability()

0. Executive Summary
LexAI is a remarkably ambitious SPA with excellent architectural intent (clean layers, provider abstraction, authority-based RBAC) but has five fundamental problems that prevent it from being a production legal SaaS:
1. No backend server — Browser talks directly to Supabase with anon key. Every security control runs client-side. Any user with devtools can bypass everything.
2. No test infrastructure — 0 tests across 200+ modules. A single regression can break the entire app silently.
3. Monolithic CSS + no design system — 27K lines of CSS in one file, no component isolation, no dark mode.
4. Simulated features — "Encryption", "compression", "OCR success" are flags that do nothing.
5. No monitoring, no offline, no performance measurement — The "Performance Analytics" page shows hardcoded values.
This plan fixes all five in four phases over ~6 months.

Key architectural decisions:
- Abandon the "no backend" SPA approach. A legal application handling PII, case documents, and court data must have a server-side enforcement layer.
- GraphQL primary, REST secondary. GraphQL gives typed queries, avoids over/under-fetching, and allows one round trip for complex pages. Apollo Client replaces queryCache.js.
- PostgreSQL via Kysely (type-safe query builder) — not raw Supabase REST. Kysely generates SQL, enforces types, and works with any pg driver.
- Redis for session cache, query cache, rate limiter counters, and pub/sub for cross-tab invalidation.
- Minio (S3-compatible) for file storage, with signed URLs. Replaces direct Supabase Storage access.
- BullMQ for background jobs (backup, OCR, sync, bulk email).

9. What NOT to Do
Avoid	Reason
Rewriting everything at once	"Big Bang" rewrite will take 12+ months and die. Incremental migration is the only path.
Porting to TypeScript before Phase 3	The JS codebase works. Adding types first adds churn with zero user benefit. Do it during Phase 3 when tests validate correctness.
Building a real-time sync engine	Offline sync is complex. Add service worker cache-first for reads in Phase 4, skip offline writes for v1.
Replacing React	React works fine. The problem is missing backend, not the frontend framework.
Microservices	The app is a legal CRM. A monolith with clear module boundaries handles 100K users. Microservices add complexity without benefit.
Keeping the "no backend" architecture	It's architecturally unsound for any app handling PII. Legal data requires server-side audit trails, access logs, and encryption at rest.
